# Customer Unit - Business Logic Model

## Overview
Customer Unit의 핵심 비즈니스 로직과 워크플로우를 정의합니다.

---

## 1. 테이블 자동 로그인 비즈니스 로직

### 1.1 초기 설정 워크플로우

```
[관리자] 테이블 태블릿 초기 설정
    ↓
[입력] 매장 ID, 테이블 번호, 테이블 비밀번호
    ↓
[검증] 백엔드 API 호출 → POST /api/auth/table-login
    ↓
[성공] JWT Access Token (16시간) + Refresh Token (30일) 발급
    ↓
[저장] LocalStorage에 저장
    - accessToken
    - refreshToken
    - storeId
    - tableId
    - tableNumber
    ↓
[리다이렉트] 메뉴 화면으로 이동
```

### 1.2 자동 로그인 워크플로우

```
[앱 시작] 페이지 로드 또는 새로고침
    ↓
[확인] LocalStorage에서 토큰 확인
    ↓
[검증] Access Token 유효성 확인
    ↓
    ├─ [유효] → 메뉴 화면 표시
    │
    ├─ [만료] → Refresh Token으로 갱신 시도
    │       ↓
    │   [API 호출] POST /api/auth/refresh
    │       ↓
    │       ├─ [성공] → 새 Access Token 발급 → 메뉴 화면 표시
    │       │
    │       └─ [실패] → 초기 설정 화면으로 이동
    │
    └─ [없음] → 초기 설정 화면으로 이동
```

### 1.3 토큰 갱신 로직

**Access Token**:
- 유효 기간: 16시간
- 용도: API 요청 인증
- 저장 위치: LocalStorage

**Refresh Token**:
- 유효 기간: 30일
- 용도: Access Token 갱신
- 저장 위치: LocalStorage

**갱신 프로세스**:
1. API 요청 시 Access Token 만료 감지 (401 Unauthorized)
2. Refresh Token으로 POST /api/auth/refresh 호출
3. 새 Access Token 발급 받음
4. LocalStorage 업데이트
5. 원래 요청 재시도

### 1.4 테이블 비밀번호 변경 시 처리

```
[로그인 시도] 저장된 정보로 자동 로그인
    ↓
[실패] 401 Unauthorized (비밀번호 불일치)
    ↓
[알림] 관리자에게 알림 전송 (선택사항)
    ↓
[대기] 비밀번호 재설정 대기 화면 표시
    ↓
[옵션 제공]
    - "재시도" 버튼: 다시 로그인 시도
    - "초기 설정" 버튼: 초기 설정 화면으로 이동
```

## 2. 메뉴 조회 및 필터링 로직

### 2.1 메뉴 로드 워크플로우

```
[페이지 로드] 메뉴 화면 진입
    ↓
[기본 카테고리] "전체" 또는 첫 번째 카테고리 선택
    ↓
[API 호출] GET /api/menus?storeId={storeId}&category={category}
    ↓
[응답 처리]
    ├─ [성공] → 메뉴 목록 표시
    │       ↓
    │   [메뉴 카드 렌더링]
    │       - 메뉴명
    │       - 가격
    │       - 설명
    │       - 이미지 (placeholder 처리)
    │       - 품절 상태 (회색 처리)
    │
    └─ [실패] → 에러 메시지 표시
```

### 2.2 카테고리 필터링 로직

```
[사용자] 카테고리 탭 클릭
    ↓
[API 호출] GET /api/menus?storeId={storeId}&category={selectedCategory}
    ↓
[응답 처리] 해당 카테고리 메뉴만 표시
    ↓
[UI 업데이트]
    - 선택된 카테고리 탭 강조
    - 메뉴 목록 갱신
```

### 2.3 메뉴 상태 처리

**품절 메뉴**:
- 표시: 회색 처리 + "품절" 배지
- 동작: 장바구니 추가 버튼 비활성화

**이미지 로드 실패**:
- 표시: 기본 placeholder 이미지
- 대체 텍스트: 메뉴명

## 3. 장바구니 관리 로직

### 3.1 장바구니 데이터 구조

```typescript
interface CartItem {
  menuId: string;
  menuName: string;
  price: number;
  quantity: number;
  imageUrl?: string;
}

interface Cart {
  items: CartItem[];
  totalAmount: number;
  lastUpdated: string; // ISO timestamp
}
```

### 3.2 메뉴 추가 로직

```
[사용자] 메뉴 카드에서 "추가" 버튼 클릭
    ↓
[확인] 품절 여부 확인
    ↓
    ├─ [품절] → 에러 메시지 표시
    │
    └─ [주문 가능]
        ↓
    [확인] 장바구니에 동일 메뉴 존재 여부
        ↓
        ├─ [존재] → 수량 +1 (최대 99개 제한)
        │
        └─ [없음] → 새 항목 추가 (수량 1)
        ↓
    [저장] LocalStorage 업데이트
        ↓
    [UI 업데이트]
        - 장바구니 아이콘 배지 (총 아이템 개수)
        - 확인 토스트 메시지
```

### 3.3 수량 조절 로직

```
[사용자] 장바구니에서 수량 증가/감소 버튼 클릭
    ↓
[증가 버튼]
    ├─ [현재 수량 < 99] → 수량 +1
    └─ [현재 수량 = 99] → 최대 수량 메시지 표시
    ↓
[감소 버튼]
    ├─ [현재 수량 > 1] → 수량 -1
    └─ [현재 수량 = 1] → 항목 삭제 확인 팝업
    ↓
[총액 재계산] totalAmount = Σ(price × quantity)
    ↓
[저장] LocalStorage 업데이트
    ↓
[UI 업데이트] 수량, 총액 표시
```

### 3.4 장바구니 비우기 로직

```
[사용자] "장바구니 비우기" 버튼 클릭
    ↓
[확인 팝업] "모든 항목을 삭제하시겠습니까?"
    ↓
    ├─ [확인] → 장바구니 초기화
    │       ↓
    │   [저장] LocalStorage 업데이트
    │       ↓
    │   [UI 업데이트] 빈 장바구니 메시지 표시
    │
    └─ [취소] → 팝업 닫기
```

### 3.5 가격 동기화 로직

**시점**: 주문 생성 시

```
[주문 생성 요청]
    ↓
[백엔드] 장바구니 메뉴 ID로 최신 가격 조회
    ↓
[비교] 클라이언트 가격 vs 서버 가격
    ↓
    ├─ [일치] → 주문 생성 진행
    │
    └─ [불일치] → 가격 변경 알림
            ↓
        [옵션 제공]
            - "최신 가격으로 주문" (권장)
            - "취소"
```

## 4. 주문 생성 워크플로우

### 4.1 주문 확인 단계

```
[사용자] 장바구니에서 "주문하기" 버튼 클릭
    ↓
[프론트엔드 검증]
    ├─ 장바구니 비어있음 → 에러 메시지
    ├─ 수량 0 이하 → 에러 메시지
    └─ 검증 통과
        ↓
[주문 확인 화면 표시]
    - 주문 메뉴 목록
    - 총 금액
    - "확정" 버튼
    - "취소" 버튼
```

### 4.2 주문 생성 로직

```
[사용자] "확정" 버튼 클릭
    ↓
[API 호출] POST /api/orders
    Request Body:
    {
      storeId: string,
      tableId: string,
      sessionId: string,
      items: [
        { menuId, quantity, price }
      ],
      totalAmount: number
    }
    ↓
[백엔드 검증]
    ├─ 메뉴 존재 여부 확인
    ├─ 가격 일치 여부 확인
    ├─ 품절 여부 확인
    └─ 검증 통과
        ↓
[주문 생성]
    - 주문 번호 생성 (날짜 + 순번: 20260209-001)
    - DynamoDB에 저장
    - SSE로 관리자에게 실시간 전송
    ↓
[응답]
    Response:
    {
      orderId: string,
      orderNumber: string,
      createdAt: string
    }
    ↓
[프론트엔드 처리]
    ├─ [성공]
    │   ↓
    │   [주문 성공 화면 표시]
    │       - 주문 번호
    │       - "주문이 접수되었습니다" 메시지
    │       - 5초 카운트다운 (수동 이동 불가)
    │   ↓
    │   [5초 후]
    │       - 장바구니 비우기
    │       - 메뉴 화면으로 자동 리다이렉트
    │
    └─ [실패]
        ↓
        [에러 처리]
            - 사용자 친화적 메시지 표시
            - 장바구니 유지
            - 재시도 옵션 제공
```

### 4.3 네트워크 오류 처리

```
[네트워크 오류 발생]
    ↓
[자동 재시도] 최대 3회
    ↓
    ├─ [재시도 성공] → 주문 생성 완료
    │
    └─ [3회 실패]
        ↓
        [에러 메시지]
            "네트워크 오류가 발생했습니다.
             잠시 후 다시 시도해주세요."
        ↓
        [옵션]
            - "재시도" 버튼
            - "취소" 버튼 (장바구니 유지)
```

## 5. 주문 내역 조회 로직

### 5.1 주문 내역 로드

```
[페이지 진입] 주문 내역 화면
    ↓
[API 호출] GET /api/orders?tableId={tableId}&sessionId={sessionId}
    ↓
[응답 처리]
    ├─ [성공]
    │   ↓
    │   [정렬] 최신 주문이 맨 위 (createdAt DESC)
    │   ↓
    │   [표시]
    │       - 주문 번호
    │       - 주문 시각
    │       - 메뉴 목록 (축약)
    │       - 총 금액
    │       - 주문 상태 (대기중/준비중/완료)
    │
    └─ [실패] → 에러 메시지 표시
```

### 5.2 무한 스크롤 로직

```
[사용자] 스크롤 하단 도달
    ↓
[확인] 더 불러올 데이터 존재 여부
    ↓
    ├─ [존재]
    │   ↓
    │   [API 호출] GET /api/orders?tableId={tableId}&sessionId={sessionId}&offset={offset}
    │   ↓
    │   [응답] 추가 주문 목록
    │   ↓
    │   [UI 업데이트] 기존 목록에 추가
    │
    └─ [없음] → "모든 주문을 불러왔습니다" 메시지
```

### 5.3 실시간 상태 업데이트

```
[SSE 연결] 페이지 로드 시 SSE 연결 생성
    ↓
[이벤트 수신] 주문 상태 변경 이벤트
    Event Data:
    {
      orderId: string,
      status: "pending" | "preparing" | "completed"
    }
    ↓
[UI 업데이트]
    - 해당 주문의 상태 배지 업데이트
    - 상태별 색상 변경
        - 대기중: 노란색
        - 준비중: 파란색
        - 완료: 초록색
```

## 6. 에러 처리 및 예외 시나리오

### 6.1 API 호출 실패 처리

```
[API 호출 실패]
    ↓
[에러 타입 확인]
    ├─ [401 Unauthorized]
    │   ↓
    │   [토큰 갱신 시도]
    │       ├─ [성공] → 원래 요청 재시도
    │       └─ [실패] → 로그인 화면으로 이동
    │
    ├─ [404 Not Found]
    │   ↓
    │   [메시지] "요청한 정보를 찾을 수 없습니다."
    │
    ├─ [500 Internal Server Error]
    │   ↓
    │   [메시지] "일시적인 오류가 발생했습니다. 다시 시도해주세요."
    │
    └─ [Network Error]
        ↓
        [메시지] "네트워크 연결을 확인해주세요."
```

### 6.2 네트워크 연결 끊김 처리

```
[네트워크 연결 끊김 감지]
    ↓
[UI 표시]
    - 화면 상단에 "오프라인" 배너 표시
    - 빨간색 배경
    ↓
[재연결 시도]
    - 5초마다 연결 상태 확인
    ↓
    ├─ [재연결 성공]
    │   ↓
    │   [UI 업데이트]
    │       - "온라인" 배너 표시 (초록색)
    │       - 3초 후 배너 자동 숨김
    │       - 데이터 재로드
    │
    └─ [재연결 실패] → 계속 재시도
```

### 6.3 세션 만료 처리

```
[세션 만료 감지]
    ↓
[Refresh Token 확인]
    ↓
    ├─ [유효]
    │   ↓
    │   [자동 재로그인]
    │       - POST /api/auth/refresh
    │       - 새 Access Token 발급
    │       - 원래 작업 계속
    │
    └─ [만료]
        ↓
        [초기 설정 화면으로 이동]
            - "세션이 만료되었습니다" 메시지
            - 저장된 정보로 자동 로그인 시도
```

## 7. 데이터 무결성 규칙

### 7.1 장바구니 무결성

**규칙**:
- 장바구니에 담긴 메뉴가 삭제된 경우, 주문 생성 시 백엔드에서 검증
- 주문 생성 실패 시 에러 메시지 표시: "일부 메뉴가 더 이상 제공되지 않습니다."

### 7.2 가격 무결성

**규칙**:
- 주문 생성 시 백엔드에서 최신 가격으로 재계산
- 가격 불일치 시 사용자에게 알림 및 선택 옵션 제공

### 7.3 재고 관리

**규칙**:
- 재고 관리 없음 (무제한 주문 가능)
- 품절 상태는 관리자가 수동으로 설정

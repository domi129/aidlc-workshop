## 1. 토큰 갱신 메커니즘 (Q1-2)

**원래 답변**: C) 토큰 갱신(refresh token) 메커니즘 사용

**문제점**: 
- 기존 요구사항에는 "JWT 토큰 기반 무상태 인증"으로 명시되어 있습니다.
- Refresh token 메커니즘은 추가 구현이 필요하며, 상태 관리가 필요합니다.

**추가 질문 CQ1-1**: Refresh token을 어떻게 구현하시겠습니까?
- A) Access token (16시간) + Refresh token (30일) 구조로 변경
- B) 단순히 저장된 테이블 정보로 자동 재로그인 (기존 A 옵션)
- C) 토큰 만료 시 사용자에게 재로그인 요청 (기존 B 옵션)
- D) 기타 (설명 필요)

[Answer]: A

---

## 2. 테이블 비밀번호 변경 시 처리 (Q1-3)

**원래 답변**: C) 자동으로 비밀번호 재설정 요청

**문제점**:
- "자동으로 비밀번호 재설정"의 의미가 불명확합니다.
- 테이블 태블릿은 고객이 사용하는 디바이스이므로, 비밀번호 재설정 권한이 없을 수 있습니다.

**추가 질문 CQ1-2**: 테이블 비밀번호 변경 시 정확히 어떻게 처리하시겠습니까?
- A) 로그인 실패 후 초기 설정 화면으로 이동하여 관리자가 새 비밀번호 입력
- B) 관리자에게 알림을 보내고 비밀번호 재설정 대기
- C) 에러 메시지만 표시하고 재시도 허용
- D) 기타 (설명 필요)

[Answer]: B

---

## 3. 장바구니 데이터 저장 위치 (Q3-3)

**원래 답변**: D) 백엔드 서버에 저장

**문제점**:
- 기존 요구사항에는 "클라이언트 측 임시 저장 (LocalStorage 또는 SessionStorage)"으로 명시되어 있습니다.
- 백엔드 저장 시 추가 API 엔드포인트와 데이터베이스 테이블이 필요합니다.

**추가 질문 CQ3-1**: 장바구니를 백엔드에 저장하는 이유와 구현 방식을 설명해주세요.
- A) 요구사항 변경: 백엔드에 저장하고 API 엔드포인트 추가 (POST /api/cart, GET /api/cart, DELETE /api/cart)
- B) 원래 요구사항 유지: LocalStorage에 저장 (기존 A 옵션)
- C) SessionStorage에 저장 (기존 B 옵션)
- D) 기타 (설명 필요)

[Answer]: B

**추가 질문 CQ3-2**: 백엔드 저장을 선택한 경우, 장바구니 데이터 구조는 어떻게 되나요?
- A) 테이블별 장바구니 (tableId + sessionId 기준)
- B) 사용자별 장바구니 (JWT 토큰 기준)
- C) 세션별 장바구니 (sessionId만 기준)
- D) 기타 (설명 필요)

[Answer]: 백엔드 저장 선택 안함

## 4. 주문 생성 검증 (Q4-1)

**원래 답변**: A) 프론트엔드에서만 검증

**문제점**:
- 보안상 백엔드 검증이 필수적입니다 (메뉴 존재 여부, 가격 조작 방지).
- 프론트엔드만 검증 시 악의적인 요청을 막을 수 없습니다.

**추가 질문 CQ4-1**: 주문 생성 검증을 프론트엔드만 수행하는 이유가 있나요?
- A) 요구사항 변경: 프론트엔드와 백엔드 모두 검증 (권장)
- B) 프론트엔드만 검증 유지 (보안 위험 감수)
- C) 백엔드에서만 검증 (프론트엔드 검증 제거)
- D) 기타 (설명 필요)

[Answer]: A

## 5. 주문 상태 실시간 업데이트 (Q5-3)

**원래 답변**: B) 주기적 폴링 (예: 5초마다)

**문제점**:
- 기존 요구사항에는 "주문 상태 실시간 업데이트 (선택사항)"으로 명시되어 있습니다.
- 관리자용 인터페이스는 SSE를 사용하는데, 고객용은 폴링을 사용하는 이유가 불명확합니다.

**추가 질문 CQ5-1**: 고객용 주문 상태 업데이트 방식을 선택한 이유는 무엇인가요?
- A) SSE를 통해 실시간 업데이트 (관리자와 동일한 방식)
- B) 주기적 폴링 유지 (5초마다)
- C) 수동 새로고침만 가능
- D) 실시간 업데이트 없음 (초기 버전에서 제외)

[Answer]: A

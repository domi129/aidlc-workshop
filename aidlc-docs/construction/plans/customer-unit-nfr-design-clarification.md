## 1. CORS 설정 (Q4-2)

**원래 답변**: B) 모든 도메인 허용 (개발 편의)

**문제점**:
- 프로덕션 환경에서 모든 도메인 허용은 보안 위험이 큽니다.
- CSRF 공격에 취약해집니다.

**추가 질문 CQ4-2**: CORS 설정을 모든 도메인 허용으로 선택한 이유가 있나요?
- A) 개발 환경만 모든 도메인 허용, 프로덕션은 화이트리스트
- B) 모든 환경에서 모든 도메인 허용 (보안 위험 감수)
- C) 특정 도메인만 허용 (화이트리스트)
- D) 기타 (설명 필요)

[Answer]: B

## 2. 입력 검증 전략 (Q4-3)

**원래 답변**: C) 프론트엔드만 검증

**문제점**:
- Functional Design의 BR-ORDER-004~007에서 백엔드 검증이 명시되어 있습니다.
- NFR Requirements의 CQ4-1에서 "프론트엔드 + 백엔드 모두 검증"으로 결정되었습니다.
- 프론트엔드만 검증 시 악의적인 API 호출을 막을 수 없습니다.

**추가 질문 CQ4-3**: 입력 검증을 프론트엔드만 수행하는 이유가 있나요?
- A) 이전 결정 유지: 프론트엔드 + 백엔드 이중 검증 (권장)
- B) 프론트엔드만 검증 유지 (보안 위험 감수)
- C) 백엔드만 검증
- D) 기타 (설명 필요)

[Answer]: B

## 3. 번들 크기 최적화 (Q3-2)

**원래 답변**: C) 최적화 없음

**문제점**:
- NFR Requirements에서 페이지 로드 시간 2초 이하 목표를 설정했습니다.
- 번들 크기 최적화 없이 2초 이하 달성이 어려울 수 있습니다.
- Material-UI는 번들 크기가 큰 라이브러리입니다.

**추가 질문 CQ3-2**: 번들 크기 최적화를 하지 않는 이유가 있나요?
- A) 초기 버전 단순화, 추후 최적화 예정
- B) Tree Shaking만 적용 (Vite 기본 제공)
- C) 코드 스플리팅 + Tree Shaking 적용 (권장)
- D) 기타 (설명 필요)

[Answer]: A

## 4. API 요청 최적화 (Q3-3)

**원래 답변**: C) 최적화 없음

**문제점**:

- 서버 부하 증가 및 비용 증가 가능성이 있습니다.

**추가 질문 CQ3-3**: API 요청 최적화를 하지 않는 이유가 있나요?
- A) 초기 버전 단순화, 추후 최적화 예정

- C) 디바운싱 적용 (검색 등)
- D) 기타 (설명 필요)

[Answer]: A

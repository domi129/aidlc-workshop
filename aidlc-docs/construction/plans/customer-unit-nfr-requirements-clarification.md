## 1. API 재시도 정책 (Q6-1)

**원래 답변**: C) 재시도 없음

**문제점**:
- Functional Design의 BR-ORDER-013에서 "네트워크 오류 시 최대 3회 자동 재시도"로 정의되어 있습니다.
- 주문 생성 워크플로우에서 자동 재시도 로직이 명시되어 있습니다.

**추가 질문 CQ6-1**: API 재시도 정책을 재시도 없음으로 선택한 이유가 있나요?
- A) Functional Design 변경: 자동 재시도 제거
- B) Functional Design 유지: 지수 백오프 재시도 (1초, 2초, 4초) 적용
- C) Functional Design 유지: 고정 간격 재시도 (1초마다 3회) 적용
- D) 기타 (설명 필요)

[Answer]: B

**추가 질문 CQ6-2**: 에러 모니터링을 사용하지 않는 이유가 있나요?
- A) 초기 버전에서는 제외, 추후 추가 예정
- B) CloudWatch Logs만 사용 (기본 로깅)
- C) Sentry 또는 CloudWatch Alarms 사용
- D) 기타 (설명 필요)

## 5. DynamoDB 클라이언트 (Q5-6)

**원래 답변**: B) AWS SDK v2 (안정적)

**문제점**:
- AWS SDK v2는 2023년부터 유지보수 모드이며, 새 프로젝트에는 v3 권장됩니다.
- v3는 모듈화되어 번들 크기가 작고 성능이 더 좋습니다.

**추가 질문 CQ5-6**: AWS SDK v2를 선택한 이유가 있나요?
- A) 기존 프로젝트와의 호환성
- B) 팀의 v2 경험
- C) AWS SDK v3로 변경 (권장)
- D) 기타 (설명 필요)

[Answer]: A

## 6. 페이지 로드 시간 vs API 응답 시간 (Q2-1, Q2-2)

**원래 답변**: 
- Q2-1: B) API 응답 시간 평균 500ms 이하
- Q2-2: A) 페이지 로드 시간 1초 이하

**문제점**:
- 페이지 로드 시간 1초 이하를 달성하려면 API 응답 시간이 200ms 이하여야 합니다.
- API 응답 500ms + 렌더링 시간 + 네트워크 지연 = 1초 초과 가능성 높음

**추가 질문 CQ2-1**: 페이지 로드 시간 목표를 조정하시겠습니까?
- A) 페이지 로드 시간 2초 이하로 조정 (현실적)
- B) API 응답 시간 200ms 이하로 조정 (도전적)
- C) 둘 다 유지 (최적화 필요)
- D) 기타 (설명 필요)

[Answer]: A
